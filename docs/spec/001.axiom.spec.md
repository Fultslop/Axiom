# Spec: JSDoc-Driven Contract Enforcement (fsprepost)

## 1. The "Why": The Failure of Existing Methods
Standard TypeScript and existing validation libraries fail to meet the "Zero-Friction" requirement for high-velocity engineering:

* **Manual `if` Checks:** Clutter function bodies, are prone to "copy-paste" errors, and require manual removal logic for production performance.
* **Assertion Libraries:** Create a "cringe" factor with verbose, non-standard syntax (e.g., `assert.isProLevel(val)`) and often leave footprints in the production bundle.
* **Decorator-Based Validation:** Only works on classes, requires experimental flag stabilization, and creates a "black box" logic flow that is hard to read in standard documentation.
* **The Documentation Gap:** In most projects, JSDoc and actual logic drift apart. A comment says `@param amount Must be positive`, but the code never checks it.

## 2. Syntax & Annotation
The transformer targets three specific JSDoc tags on **public** members:

| Tag | Placement | Scope | logic |
| :--- | :--- | :--- | :--- |
| `@pre` | Method/Function | Parameters, `this` | Must be true before body execution. |
| `@post` | Method/Function | Parameters, `this`, `result` | Must be true after body execution. |
| `@invariant` | Class | `this`, Class members | Must be true before and after every public method. |

---

## 3. Execution & Visibility Rules
To maintain the "internal flexibility vs. external rigidity" balance:

1.  **Public Boundary Only:** Contracts are only injected into `public` methods or default-visibility functions.
2.  **Private Protection:** If `@pre` or `@post` are detected on `private` or `protected` methods, the compiler **emits a warning** and skips injection.
3.  **Inheritance:** Classes inheriting from a parent with `@invariant` tags will automatically include the parent's invariants in their own public method checks.

---

## 4. Technical Transformation (The "Dev Build")
During compilation with `ts-patch`, the following AST modifications occur:

### A. The Synchronous Wrap
For standard methods, logic is injected in a "Guard Rail" pattern:
1.  **Entry Invariants:** Verify object state.
2.  **Pre-conditions:** Verify inputs.
3.  **Core Logic:** Captured in a temporary scope to extract `result`.
4.  **Post-conditions:** Verify output.
5.  **Exit Invariants:** Verify object state hasn't been corrupted.

### B. The Asynchronous Wrap (Node 25 Optimized)
For `async` methods or functions returning `Promise`, the transformer wraps the body to ensure `@post` and `@invariant` checks await the resolution of the promise before firing.



---

## 5. Failure Mode: `ContractViolationError`
When a contract is violated in Development, the system throws a structured error to ensure the developer can fix it immediately.

```typescript
{
  type: "PRE-CONDITION" | "POST-CONDITION" | "INVARIANT",
  expression: "this.balance >= 0",
  location: "AccountService.withdraw",
  stack: "..." // Points to the caller of the method, not the internal check
}
```

---

## 6. Zero-Footprint Production (The "Release Build")
This is the core requirement. fsprepost utilizes a **dual-path compilation strategy**:

* **Dev Path:** Uses `tspc` (ts-patch) to inject the transformer logic.
* **Release Path:** Uses standard `tsc`. Because standard TypeScript treats JSDoc as inert comments, the resulting JavaScript is **pure, naked logic**. 

> **Result:** There is 0% performance overhead and 0 bytes of contract code in the production environment.

---

## 7. Constraints & Edge Cases
* **Expression Support:** JSDoc comments must be valid JavaScript expressions.
* **No Side Effects:** Contracts should be "Read-Only." The transformer does not support expressions that modify state (e.g., `@pre amount++ > 0` is forbidden by convention).
* **Node 25 Compatibility:** Handles native ESM modules and private class fields (`#private`) without syntax corruption.

---

## 8. Summary of Developer Friction
* **Writing Code:** Write standard JSDoc (something you should be doing anyway).
* **Running Code:** Run your app. If you break a rule, you get a clear error.
* **Shipping Code:** No changes needed. The build pipeline handles the removal.

### 2. Why Existing Solutions Fall Short

#### A. The "Zod" Problem: Runtime Dead Weight
Zod is excellent for validating external data (API responses), but using it for internal pre/post conditions is "cringey" because:
* **The Footprint:** Zod schemas are bundled into production. In a large app, thousands of internal contract checks significantly increase bundle size.
* **The Verbosity:** You must define a schema *and* a type, then call `.parse()`. This creates 5-10 lines of "guard" code for every 1 line of actual business logic.
* **fsprepost Advantage:** fsprepost checks only exist in your `dist/` during development. In production, they evaporate.

#### B. The "Effect" Problem: Paradigm Lock-in
Effect is a powerful engine, but it requires "buying into the church of Effect."
* **The Learning Curve:** To get pre/post conditions in Effect, you must wrap everything in `Effect.gen`. Your team no longer writes "TypeScript"; they write "Effect-TypeScript."
* **Interoperability:** Mixing Effect code with standard `async/await` code creates significant friction and "wrapper hell."
* **fsprepost Advantage:** fsprepost is **transparent**. You continue writing standard, idiomatic TypeScript. The contracts are an "invisible" layer that assists you without demanding you change your coding style.

#### C. The "Decorator" Problem: Static and Heavy
Standard Stage 3 Decorators (like those in NestJS) are a step closer but:
* **Dependency:** They require runtime support.
* **Opaque:** They don't show up in standard JSDoc/IntelliSense tooltips as clearly as a `@pre` tag does.
* **fsprepost Advantage:** By using JSDoc, you get **Automated Documentation**. Any developer hovering over your function in their IDE sees the constraints immediately without needing to know a decorator library.

---

### 3. The "Invisibility" Requirement
Existing methods focus on **Runtime Safety**. fsprepost focuses on **Developer Ergonomics**. 

> **The Core Thesis:** A contract is a promise made during development. If a contract is violated, it is a developer bug, not a user error. Therefore, the code to check that contract has no business being in the user's browser or the production server's memory.

---

### 4. Strategic Justification for fsprepost

1.  **Enforced Documentation:** You cannot document a constraint without enforcing it, and you cannot enforce it without documenting it. This eliminates "stale docs."
2.  **Performance Parity:** Because the production build uses standard `tsc`, the output is as fast as if you had written no checks at all.
3.  **Low Barrier to Entry:** A junior developer can understand `@pre amount > 0` instantly. They might take weeks to master `Effect.filterOrElse`.
4.  **Modern Compatibility:** Fully supports **Node 25** features like native ESM and private class fields, which often break older AOP (Aspect Oriented Programming) libraries.

---

### 5. Conclusion of Comparison
fsprepost is not a replacement for Zod (which should still be used at the "Dirty Edge" for API input). Instead, fsprepost is the **Internal Integrity Layer**. It provides the safety of Effect and the validation of Zod with **0% of the runtime cost** and **10% of the friction**.
