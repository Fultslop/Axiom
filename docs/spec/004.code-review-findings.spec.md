# Spec 004: Code Review Findings

Issues identified during code review. To be addressed after the current cleanup
branch (`chore_RC_001_clean_up`) is merged.

**Workflow context:** Human and agent co-author the spec and define interface
contracts. Human reviews and approves (or iterates) before implementation
starts. Once agreed, the AI implements against human-approved contracts.
Contracts are the executable encoding of the human sign-off — they make
implementation drift detectable by the test suite rather than by code review.

---

## 1. Silent contract dropping on transformer error

**Severity: Critical**

### Problem

In `transformer.ts`, `tryRewriteFunction` and `tryRewriteClass` catch internal
errors and return the original, unmodified node. A `@pre` or `@post` tag that
triggers a transformer bug is silently dropped — no compile failure, no visible
indication. The user ships code they believe is protected by contracts that are
not there.

### Proposed fix

Add a strict mode option (e.g. `{ strict: true }` in plugin config) that
re-throws on internal transformer error instead of recovering. Default can
remain permissive to avoid breaking existing users, but strict mode should be
recommended in the docs and used in CI.

```typescript
// permissive (current default)
catch (err) {
  ctx.warn(`[axiom] Internal error ...`);
  return node;
}

// strict mode
catch (err) {
  throw new Error(`[axiom] Internal error ... Use strict: false to recover.`);
}
```

## 2. `isolatedModules` mismatch is a user footgun

**Severity: High**

### Problem

`tsconfig.json` sets `isolatedModules: true`. Interface contract resolution
requires full program context (cross-file type information), which is unavailable
when `isolatedModules` is active. Tests work because `jest.config.ts` overrides
this. A user whose project already uses `isolatedModules: true` (common with
Vite, esbuild, SWC-based toolchains) will get silently broken interface contract
inheritance — no error, contracts just don't propagate from interfaces.

### Proposed fix

1. Document the constraint prominently in the README under "Known Limitations".
2. Emit a warning from the transformer at startup when `isolatedModules` is
   detected in the host `CompilerOptions` and interface contracts are present.
3. Consider splitting `tsconfig.json` into `tsconfig.base.json` (library source)
   and `tsconfig.isolatedModules.json` (compatibility test target) to make the
   distinction explicit.

## 3. Branch coverage floor too low for an AST transformer

**Severity: Medium**

### Problem

Jest is configured with an 80% branch coverage threshold. Current branch
coverage is 85%. For a library that manipulates TypeScript compiler internals,
uncovered branches are disproportionately likely to be in complex edge-case
paths — multiple inheritance levels, async generators, partial property chains.
These are exactly the scenarios where users spend hours debugging unexpected
behaviour.

### Proposed fix

Raise the branch coverage threshold to 95% in `jest.config.ts`:

```typescript
coverageThreshold: {
  global: { branches: 95, functions: 90, lines: 90, statements: 90 }
}
```

Identify and cover the remaining uncovered branches in:
- `tag-pipeline.ts` (~87% branch coverage)
- `class-rewriter.ts` (merge and multi-inheritance paths)
- `assertions.ts` (JSON fallback in `deepSnapshot`)
- `require-injection.ts` (~79% branch coverage)

## 4. `deepSnapshot` fallback is silently lossy

**Severity: Low**

### Problem

`assertions.ts` falls back to `JSON.parse(JSON.stringify(obj))` when
`structuredClone` is unavailable. This silently drops `undefined` values,
`Symbol` keys, functions, and circular references. A `@prev` snapshot of an
object with `undefined` properties will differ from the original in ways the
user does not expect.

### Proposed fix

Add a warning when the JSON fallback is used:

```typescript
ctx.warn('[axiom] structuredClone unavailable; deepSnapshot uses JSON fallback. ' +
  'undefined, Symbol, and circular values will not be captured.');
```

Or restrict fallback use and document the Node.js version requirement
(`structuredClone` available since Node 17).

## 5. `exports.` prefix for module-level references breaks ESM output

**Severity: Critical**

### Problem

When a contract expression references a module-level exported name (e.g. a
`const` or `enum`), `substituteContractIdentifiers` in `ast-builder.ts` rewrites
the identifier to `exports.Name`. This is a CJS-specific pattern — it produces
`exports.MAX_LIMIT` in the emitted code. In ESM output (`module: ESNext`,
`Node16`, `NodeNext`), `exports` is not defined at runtime, so the generated
contract guard throws a `ReferenceError` at the exact moment the user expects
their contract to protect them.

The `require-injection.ts` module already knows the difference between CJS and
ESM and emits `import` statements for ESM targets. But the `exports.` prefix
rewriting has no such awareness — it always emits `exports.`, regardless of
module kind.

### Proposed fix

Thread the `moduleKind` (or a derived `isEsm` flag) through `TransformerContext`
into `buildPreCheck` / `buildPostCheck` / `substituteContractIdentifiers`. For
ESM targets, emit the bare identifier instead of `exports.Name` — the ESM
import declaration already makes the name available in scope.

## 6. `renameIdentifiersInExpression` uses regex word-boundary replacement — fragile and order-dependent

**Severity: High**

### Problem

`interface-resolver.ts` renames parameter identifiers in contract expressions
using `new RegExp('\\b' + escaped + '\\b', 'g')`. This has two failure modes:

1. **Order dependence:** If the rename map contains both `val` → `value` and
   `value` → `amount`, the result depends on `Map` iteration order. If `val`
   is processed first, the `value` replacement then also rewrites the
   *just-substituted* `value` to `amount`, producing double-renamed garbage.

2. **False boundary matches:** The `\b` word boundary considers `$` and `_` as
   word characters. A parameter named `x` would match inside `x_val` at the
   boundary between `_` and `x` if the regex is applied to the full expression
   — though this is unlikely in practice, it's a latent bug.

### Proposed fix

Parse the expression into an AST, rename at the identifier-node level, and
print back. This is what the transformer already does for contract guard
generation — the infrastructure exists in `parseContractExpression` and
`reifyExpression`. Alternatively, sort the rename map entries longest-first to
mitigate the order-dependence, and add a negative lookahead/lookbehind for `_`
and `$` to the regex.

## 7. `node.pos` as reparsed-index key is unreliable across source-file variants

**Severity: High**

### Problem

`reparsed-index.ts` maps `node.pos` → reparsed node. This position is the
*leading trivia start* of the node in the original source file. The same
function at the same position in a re-parsed file will have the same `pos`, so
the lookup works — but only as long as the source text is identical. If the
transformer ever modifies the source text before re-parsing (currently it
doesn't, but the architecture doesn't prevent it), positions would shift and
the index would return stale or missing entries silently.

More concretely: `node.pos` can collide. Two nodes that start at the same
byte offset in different source-file variants (e.g. after a previous
transformation pass) would map to the same key. The `Map` would silently
overwrite the earlier entry.

### Proposed fix

Use a composite key that includes both `pos` and a node-kind discriminator, or
switch to `node.getStart()` (which skips leading trivia) for more stable
positioning. Document the invariant that re-parsing must happen on the original
source text before any mutations.

## 8. `console` in `GLOBAL_IDENTIFIERS` whitelist is a security hole

**Severity: Medium**

### Problem

`contract-validator.ts` includes `'console'` in the `GLOBAL_IDENTIFIERS`
whitelist, allowing contract expressions like `@pre console.log(secret)`. This
means a contract expression can perform arbitrary side effects — logging
sensitive data, mutating global state via `console.log` overrides, etc. — every
time the function is called. Contracts are supposed to be pure boolean
predicates. Allowing `console` also contradicts the project's own ESLint rule
that bans `console` usage.

### Proposed fix

Remove `'console'` from `GLOBAL_IDENTIFIERS`. If a user genuinely needs
`console` in a contract expression, they can add it via the `allowIdentifiers`
option. This makes side-effect-by-contract an explicit opt-in rather than a
silent default.

## 9. Duplicated `transformWithProgram` helpers across test files

**Severity: Medium**

### Problem

Three test files define their own `transformWithProgram` or `fullProgramMode`
helper instead of importing from `test/helpers.ts`:

- `test/bug-repro.test.ts` — defines `fullProgramMode`
- `test/property-chain-runtime.test.ts` — defines `transformWithProgram`
- `test/acceptance.test.ts` — defines `compileWithTransformer`

All three are functionally identical to `transformWithProgram` in
`test/helpers.ts`. This violates DRY and creates drift risk — if the canonical
helper is updated (e.g. to support `keepContracts`), the copies won't get the
fix.

Additionally, `bug-repro.test.ts` contains `console.log()` debug output that
was left in after development.

### Proposed fix

1. Delete the duplicated helpers and import from `test/helpers.ts`.
2. Extend `transformWithProgram` in `helpers.ts` to accept the same options as
   `transform()` (i.e. `TransformOptions` instead of just `warn` and
   `mismatchMode`).
3. Remove the `console.log` statements from `bug-repro.test.ts`.

## 10. `snapshot()` is a shallow spread that silently drops prototype chain and non-enumerable properties

**Severity: Medium**

### Problem

`assertions.ts` implements `snapshot()` as `{ ...obj }`. This silently drops:
- Methods and properties on the prototype chain
- Non-enumerable properties (e.g. `length` on arrays, built-in methods)
- Symbol-keyed properties
- Property descriptors (getters become evaluated values)

A user who writes `@prev` (which defaults to `snapshot(this)`) on a class with
getter properties or inherited methods will get a snapshot that differs from the
original in ways they don't expect. The `@post` check then compares against a
corrupted `prev`, producing either false positives (spurious violations) or
false negatives (missed violations).

### Proposed fix

Document the limitation clearly in the `@prev` documentation. Consider
providing a `deepSnapshot`-style variant that uses `Object.getOwnPropertyDescriptors`
and `Object.create` to preserve the prototype chain, or at minimum warn when
`snapshot` is used on objects with getter properties.

## 11. No test coverage for `snapshot()` or `deepSnapshot()` runtime behavior

**Severity: Medium**

### Problem

`test/assertions.test.ts` tests `pre()` and `post()` but never tests `snapshot()`
or `deepSnapshot()`. These are public API functions that users call directly or
that the transformer injects via `@prev`. The JSON fallback path in
`deepSnapshot` (finding 4 from the first review) has zero test coverage.

### Proposed fix

Add tests for:
- `snapshot()` preserves own enumerable properties
- `snapshot()` does *not* preserve prototype methods or non-enumerable props
- `deepSnapshot()` with `structuredClone` available
- `deepSnapshot()` JSON fallback drops `undefined` values
- `deepSnapshot()` JSON fallback drops circular references (throws or truncates)

## 12. `package.json` declares `"type": "commonjs"` but `tsconfig.json` uses `"module": "node16"`

**Severity: Low**

### Problem

`package.json` sets `"type": "commonjs"`, which means Node treats `.js` files
as CJS. But `tsconfig.json` sets `"module": "node16"`, which tells TypeScript
to emit `require()` for CJS *or* `import` for ESM depending on the `.mts`/`.mjs`
extension or `"type"` field. Since `"type"` is `"commonjs"`, `tsc` emits CJS
`require()` — so it works today. But the README instructs users to set
`"type": "module"` in their own `package.json`. If a user copies the project's
`tsconfig.json` settings without also setting `"type": "module"`, they'll get
CJS output with ESM-style import declarations that crash at runtime.

The README's installation section says "Declare your project as ESM in
`package.json`" but doesn't explain *why* or what breaks if you don't.

### Proposed fix

1. Add a troubleshooting section to the README explaining the `"type": "module"`
   requirement and the failure mode (runtime `require is not defined` in ESM
   output).
2. Consider adding a startup warning in the transformer when `moduleKind` is
   ESM but the emitted runtime would be CJS (or vice versa).

## 13. `@invariant` on non-class nodes warns but `@pre`/`@post` on non-exported functions silently drops

**Severity: Low**

### Problem

The transformer emits a warning when `@invariant` appears on a non-class node
(function, variable, interface). But when `@pre`/`@post` appear on a
non-exported function declaration (not inside a class), the contracts are
silently dropped with no warning. The only warning path is for arrow functions
and function expressions — `emitUnsupportedFunctionWarning` is called for those,
but plain unexported `function` declarations are silently skipped.

A user who adds `@pre` to a helper function and forgets to export it will have
their contract silently ignored — the exact same class of bug as finding 1
(silent contract dropping), but caused by user error rather than a transformer
bug.

### Proposed fix

Emit a warning when `@pre`/`@post` tags are found on a non-exported function
declaration, similar to the `@invariant` misuse warning. The warning should say
something like: `@pre/@post on non-exported function 'foo' has no effect —
contracts are only injected on exported functions and public class methods`.
