# Spec 002: fsprepost MVP

## 1. Goal

Prove that a TypeScript compiler transformer can:

1. Read `@pre` and `@post` conditions from JSDoc comments.
2. Inject runtime assertion code into the compiled output during development.
3. Produce zero contract code in the production build, with no extra stripping step.

Success is defined as: the `withdraw` example below compiles, throws `ContractViolationError` when a condition is violated in dev, and produces naked function bodies in prod.

---

## 2. MVP Scope

### 2.1 Supported Tags

| Tag | Placement | Scope available in expression |
| :--- | :--- | :--- |
| `@pre` | Function or method | Named parameters, `this` (if method) |
| `@post` | Function or method | Named parameters, `this` (if method), `result` |

### 2.2 Supported Targets

- Synchronous functions and methods only.
- Public methods and exported module-level functions only.
- Plain class fields and `this` references are valid in expressions.

### 2.3 Build Paths

| Path | Command | Behavior |
| :--- | :--- | :--- |
| Dev | `tspc -p tsconfig.dev.json` | Transformer active, contracts injected |
| Release | `tsc -p tsconfig.json` | Transformer absent, JSDoc treated as inert |

No env-var switching. Two explicit tsconfig files. The developer chooses the path.

## 3. Proof-of-Concept Target

```typescript
class Account {
  balance: number = 100;

  /**
   * @pre amount > 0
   * @pre amount <= this.balance
   * @post result === this.balance
   */
  withdraw(amount: number): number {
    this.balance -= amount;
    return this.balance;
  }
}
```

Expected dev-build output (illustrative, not literal):

```typescript
withdraw(amount: number): number {
  // pre-conditions
  if (!(amount > 0))
    throw new ContractViolationError("PRE", "amount > 0", "Account.withdraw");
  if (!(amount <= this.balance))
    throw new ContractViolationError("PRE", "amount <= this.balance", "Account.withdraw");

  // body
  const result = (() => {
    this.balance -= amount;
    return this.balance;
  })();

  // post-conditions
  if (!(result === this.balance))
    throw new ContractViolationError("POST", "result === this.balance", "Account.withdraw");

  return result;
}
```

Expected release-build output:

```typescript
withdraw(amount: number): number {
  this.balance -= amount;
  return this.balance;
}
```

## 4. Expression Scope Model

The transformer evaluates expressions as JavaScript boolean expressions. Scope is:

| Identifier | Bound to |
| :--- | :--- |
| Parameter names | The actual argument values at call time |
| `this` | The method's `this` context (methods only; undefined for free functions) |
| `result` | The return value of the body (available in `@post` only) |

Expressions are injected as-is into the AST at the call site. They are **not** `eval`-ed strings at runtime — they are compiled into the output JavaScript. This means:

- They are type-checked by the compiler (a type error in a `@pre` expression is a compile error).
- They have access to the same imports and globals as the surrounding file.

## 5. ContractViolationError

```typescript
class ContractViolationError extends Error {
  constructor(
    public readonly type: "PRE" | "POST",
    public readonly expression: string,
    public readonly location: string,
  ) {
    super(`[${type}] Contract violated at ${location}: ${expression}`);
    this.name = "ContractViolationError";
  }
}
```

This class is exported from the library. The transformer imports it at the top of any file it modifies.

Stack traces point to the injection site inside the method. Caller-level stack rewriting is **not** in MVP scope.

---

## 6. Safety Invariant

If the transformer encounters any condition it cannot handle, it **must**:

1. Leave the AST node unmodified.
2. Emit a `ts.DiagnosticCategory.Warning` message identifying the file, line, and reason.
3. Never throw or halt compilation.

This means a partially-annotated codebase always compiles. Worst case: a contract is silently skipped with a warning. Never: a contract is silently skipped without a warning, or output is corrupted.

## 7. Explicit Limitations (Deferred)

The following are **known limitations of the MVP**, not bugs. Each will be addressed in a future spec.

| Limitation | Risk if ignored | Future spec |
| :--- | :--- | :--- |
| No `async`/`Promise` support | `@post` fires before promise resolves | 003 |
| No `@invariant` | Class-level state not enforced | 004 |
| No inheritance of invariants | Subclass contracts incomplete | 004 |
| No `#private` field access in expressions | Private state unverifiable | 005 |
| No warning on `@pre`/`@post` on private methods | Silent no-op | 003 |
| No mutation detection in expressions | Developer can write `@pre amount++ > 0` | Convention only |
| Stack trace points to injection site, not caller | Slightly harder to locate the bug | 005 |

---

## 8. Out of Scope (Permanently)

- Enforcing side-effect-free expressions. If a developer writes a mutating expression, that is their responsibility. The transformer does not attempt to detect or prevent it.
- Runtime schema validation of external data (use Zod for that).
- Support for JavaScript files (`.js`). TypeScript only.

## 9. Minimal Project Configuration

**`tsconfig.dev.json`**
```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "plugins": [{ "transform": "fsprepost/transformer" }]
  }
}
```

**`tsconfig.json`** — standard TypeScript config, no plugins.

The `prepublishOnly` script uses `tsc` (release path). A separate `build:dev` script uses `tspc`.

## 10. Acceptance Criteria

- [ ] `Account.withdraw(-1)` throws `ContractViolationError` with `type: "PRE"` and `expression: "amount > 0"` in dev build.
- [ ] `Account.withdraw(50)` completes without error in dev build.
- [ ] Release build of `Account.withdraw` contains no references to `ContractViolationError` or any condition check.
- [ ] A function with a malformed `@pre` expression emits a compiler warning and compiles without the contract injected.
- [ ] A `@post` expression uses `result` correctly for the synchronous case.
