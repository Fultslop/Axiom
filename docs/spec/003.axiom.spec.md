# Spec 003: Class Invariants

## 1. Goal

Add `@invariant` support: class-level conditions that are verified after every
public method exits and after the constructor exits.

---

## 2. Changes to the Error Hierarchy

Introduce `ContractError` as a shared abstract base so callers can catch all
contract failures with a single `instanceof` check.

```
Error
└── ContractError                 ← new exported base class
      ├── ContractViolationError  ← existing (@pre / @post)
      └── InvariantViolationError ← new (@invariant)
```

### 2.1 `ContractError` (new)

```typescript
export abstract class ContractError extends Error {
  public abstract readonly expression: string;
  public abstract readonly location: string;
}
```

### 2.2 `ContractViolationError` (updated — extends `ContractError`)

```typescript
export class ContractViolationError extends ContractError {
  public readonly type: 'PRE' | 'POST';
  public readonly expression: string;
  public readonly location: string;

  constructor(type: 'PRE' | 'POST', expression: string, location: string) {
    super(`[${type}] Contract violated at ${location}: ${expression}`);
    this.name = 'ContractViolationError';
    this.type = type;
    this.expression = expression;
    this.location = location;
  }
}
```

### 2.3 `InvariantViolationError` (new)

```typescript
export class InvariantViolationError extends ContractError {
  public readonly expression: string;
  public readonly location: string;

  constructor(expression: string, location: string) {
    super(`[INVARIANT] Invariant violated at ${location}: ${expression}`);
    this.name = 'InvariantViolationError';
    this.expression = expression;
    this.location = location;
  }
}
```

---

## 3. Syntax

`@invariant` tags are placed on the **class** JSDoc comment, not on individual
methods.

```typescript
/**
 * @invariant this.balance >= 0
 * @invariant this.owner !== null
 */
class BankAccount {
  balance: number;
  owner: string;

  constructor(owner: string, initial: number) { ... }

  /**
   * @pre amount > 0
   * @pre amount <= this.balance
   */
  withdraw(amount: number): void { ... }

  deposit(amount: number): void { ... }
}
```

Multiple `@invariant` tags are allowed; each is a separate check.

---

## 4. Semantics

### 4.1 When checks fire

| Site | Check |
| :--- | :--- |
| Constructor exit | All invariants (exit-only — object is being initialised) |
| Public method exit | All invariants (after `@post` checks, before `return`) |
| Private / protected method | Not checked |
| Static method | Not checked (no instance `this`) |
| Getter | Not checked (read-only, no state mutation) |
| Setter | Checked at exit (mutates state) |

### 4.2 Ordering within a method

```
ENTRY: @pre checks
BODY:  (captured via IIFE when @post or @invariant present)
EXIT:  @post checks → invariant checks → return result
```

---

## 5. Generated Output

The transformer adds a single private method `#checkInvariants` to the class
and calls it from each applicable site. Requires **ES2022+** target (private
class fields).

```typescript
// Source
/**
 * @invariant this.balance >= 0
 */
class BankAccount {
  balance = 100;

  /** @pre amount > 0 */
  withdraw(amount: number): number {
    this.balance -= amount;
    return this.balance;
  }

  constructor(initial: number) {
    this.balance = initial;
  }
}

// Transformed (illustrative)
class BankAccount {
  balance = 100;

  #checkInvariants(location: string): void {
    if (!(this.balance >= 0))
      throw new InvariantViolationError('this.balance >= 0', location);
  }

  withdraw(amount: number): number {
    if (!(amount > 0))
      throw new ContractViolationError('PRE', 'amount > 0', 'BankAccount.withdraw');
    const result = (() => {
      this.balance -= amount;
      return this.balance;
    })();
    this.#checkInvariants('BankAccount.withdraw');
    return result;
  }

  constructor(initial: number) {
    this.balance = initial;
    this.#checkInvariants('BankAccount.constructor');
  }
}
```

Note: when a method has **no** `@post` tags but the class has `@invariant` tags,
the body is still wrapped in an IIFE so that `#checkInvariants` fires after the
body completes and `result` is captured for the return.

Actually — if there are no `@post` tags, there is no need for IIFE capture.
The invariant call is simply appended as the last statement before any `return`.
See §6 for the body-with-no-post case.

---

## 6. Body Transformation Rules

### 6.1 Method has `@post` tags (existing behaviour, unchanged)

Body is captured in IIFE → `@post` checks run → `#checkInvariants` → `return result`.

### 6.2 Method has `@invariant` on class but no `@post` tags

The body statements are emitted as-is. `#checkInvariants` is injected immediately
before the final `return` statement (or appended if the method has no explicit
`return`).

> **Problem:** a method may have multiple return paths. Injecting before the
> final return is insufficient.

**Resolution:** treat the presence of `@invariant` as equivalent to requiring
post-check wrapping. The body is always captured in an IIFE when the class has
invariants, even if the method has no `@post` tags. This keeps the
transformation uniform and avoids multi-return-path analysis.

---

## 7. Clash Detection

If the class already defines a member named `#checkInvariants`, the transformer
must:

1. Emit a warning identifying the class and file.
2. Skip invariant injection for that class entirely.
3. Never corrupt the AST.

---

## 8. Scope Available in `@invariant` Expressions

| Identifier | Bound to |
| :--- | :--- |
| `this` | The class instance |
| Plain identifiers | Module-level names visible in the file |

Parameter names and `result` are **not** in scope (invariants are class-level,
not method-level).

---

## 9. Required Configuration Change

`tsconfig` must target ES2022 or later for `#` private methods to emit
correctly.

```json
{ "compilerOptions": { "target": "ES2022" } }
```

## 11. Acceptance Criteria

- [ ] A class with `@invariant this.balance >= 0` throws `InvariantViolationError`
      when a public method leaves `balance` negative.
- [ ] `catch (e instanceof ContractError)` catches both `ContractViolationError`
      and `InvariantViolationError`.
- [ ] The invariant fires after the constructor sets an invalid initial state.
- [ ] Private and static methods do **not** trigger the invariant.
- [ ] A class without `@invariant` is unaffected (no `#checkInvariants` emitted).
- [ ] A class that already defines `#checkInvariants` produces a warning and
      compiles without invariant injection.
- [ ] `@pre` / `@post` on a method in an `@invariant` class work correctly;
      invariant fires after `@post`.
- [ ] Release build contains no `#checkInvariants`, no `InvariantViolationError`.
